ClinicalTrials.gov Query → Visualization Agent

Architecture & design · natural-language question → structured visualization spec
TypeScript + Express OpenAI (structured output) + validation ClinicalTrials.gov API v2 SSE streaming Deep citations

1. One-line summary

A user asks about clinical trials in natural language → the LLM decides only "how to query and visualize" (a QueryPlan) → deterministic code fetches data from the real ClinicalTrials.gov API → and returns a structured visualization spec (with source citations) a frontend can render without guessing. The agent first decides whether a chart is even possible; if not, it returns no chart with an explanation instead.

2. End-to-end pipeline

The core idea is a clean split between the interpretation (LLM) step and the data & computation (deterministic code) steps.

IN
Natural-language query
{ "query": "How are melanoma trials distributed across phases?", "condition": "melanoma" }  — via POST /query or GET /query/stream (SSE)
1
interpret — understand the query
OpenAI structured output (JSON Schema) produces a QueryPlan; with no key, a deterministic keyword parser is used instead. → decides which filters / grouping / chart type / status to use (never produces data).
2
finalizePlan — validate & gate
Zod validation + caller-supplied fields override inference + per-viz-type constraints + ranking (sort/topN) normalization. Decides status: if unsupported/needs_clarification (incl. a zero-filter guard) → return early with no chart.
3
buildQueryParams — build API params
QueryPlan.filters → v2 query (query.cond/intr/spons/locn, Essie filter.advanced for phase/date/studyType, filter.overallStatus)
4
fetchStudies — fetch real data
Paginates ALL matches by default (no truncation → exact counts; optional MAX_STUDIES cap) + retry/backoff + totalCount. Per-page progress drives the SSE fetch_progress event.
5
normalize — flatten records
Complex raw JSON → analysis-friendly NormalizedStudy (nctId, phases, startYear, sponsor, conditions, interventions, countries, enrollment, officialTitle, investigator…)
6
aggregate — compute + attach evidence
Runs one of count / time-series / histogram / scatter / network. Every data point carries deep citations (NCT id + exact excerpt + link).
7
assemble — build response
VisualizationSpec (type·title·encoding·data) + ResponseMeta + references[] (source cards) + trace (plan + raw API I/O)
OUT
AgentResponse (JSON)
A structured response a frontend can render without guessing
LLM-involved step (only one) Deterministic code (real data & numbers)

3. Key design decision: the LLM never produces data

The model only ever emits a QueryPlan (how to query and visualize).
Every number in the response is computed from real ClinicalTrials.gov records, so the model cannot fabricate trial counts, NCT ids, or results. This is the central anti-hallucination guard, directly addressing the rubric's "avoid hallucination-prone steps / include validation".

6. Anti-hallucination & safety guards

① LLM can't produce data
Emits only a QueryPlan → cannot fabricate numbers or NCT ids
② Structured output + schema validation
JSON Schema enforced, then re-validated with Zod; enum controlled vocabularies
③ Constraint layer (planner)
Repairs/downgrades invalid plans, records reasons in warnings
④ "Is a chart even possible?" gate
status = unsupported / needs_clarification → visualization: null + explanation; zero-filter queries are guarded
⑤ Exact counts, honest limits
Full fetch by default (truncated=false); top-N category cap & excluded-record counts are always reported
⑥ Full traceability
meta.apiRequests + trace (plan + raw API sample); every datum cited

7. Input / output schema

Formal contract for all endpoints: openapi.yaml (OpenAPI 3.1). Human-readable summary below.

Request (POST /query)

{
  "query": "...",        // required
  "drug_name": "...",    // the rest optional
  "condition": "...",
  "sponsor": "...",
  "country": "...",
  "trial_phase": "PHASE3",
  "status": ["RECRUITING"],
  "study_type": "INTERVENTIONAL",
  "start_year": 2015,
  "end_year": 2025,
  "visualization_type": "bar_chart"
}

Zod .strict() rejects unknown fields; invalid requests get 400 + per-field reasons. Structured fields override LLM inference.

Response

{
  // null when status != "ok"
  "visualization": {
    "type": "bar_chart",
    "title": "...",
    "encoding": { "x":{"field":"category"},
                  "y":{"field":"trial_count"} },
    "data": [{ "category":"Phase 3", "trial_count":41,
              "citations":[{"nct_id":"NCT...",…}] }]
  },
  "meta": { "status":"ok", "totalMatchingTrials":3716,
            "analyzedTrials":3716, "truncated":false,
            "interpreter":"llm", "confidence":0.9,
            "apiRequests":["https://..."], "warnings":[…] },
  "references": [{ "nct_id", "title", "officialTitle",
                  "sponsor", "investigator", "startDate", "url" }],
  "trace": { "queryPlan", "apiRequests", "apiTotalCount",
             "apiResponseSample" }
}

8. When no chart is produced (the status gate)

The agent first decides whether the question is answerable from trial-registry data:
  • ok → produce a chart.
  • needs_clarification — on-topic but too vague (e.g. "show me trials", or a zero-filter query that would scan the whole registry) → visualization: null, meta.notes asks for specifics. The demo shows an interactive "narrow your search" form.
  • unsupported — out of scope (e.g. patient counts, drug prices, medical advice) → visualization: null, meta.notes explains why.
This prevents forcing a misleading chart, and the zero-filter guard also blocks runaway full-registry fetches.

9. Comparison (grouped bar): multi-query strategy

For "Drug A vs Drug B", mixing both entities into one query would make the counts inseparable. Instead, a separate API query runs per compared entity; each result is aggregated by phase and merged into one grouped-bar dataset via a series field (pipeline.ts → executeComparison).

10. Streaming & observability

SSE (GET /query/stream) streams the run as events status → interpretation → fetch_progress → result → done, so a client can show the agent's plan and live fetch progress before the final result. Observability is optional: when keys are present, each request is traced in Langfuse (LLM token/cost/latency) with deterministic quality scores, and OpenTelemetry spans can be exported — all no-ops without configuration.
