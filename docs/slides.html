Technical design deck

The problem

  • User asks in natural language: "How are melanoma trials distributed across phases?"
  • System must: interpret → fetch real data → pick a chart → emit a render-ready spec
  • Output must be structured & trustworthy — a frontend renders it without guessing.
Hard constraint in a clinical domain: numbers must be exact and traceable. An LLM that "makes up" trial counts is unacceptable.

Core design decision

The LLM interprets. The code computes.

The model emits only a QueryPlan (which filters, grouping, chart type). Every number comes from real CT.gov records — computed deterministically.

→ The model literally cannot fabricate counts, NCT ids, or results. This single rule shapes the whole architecture.

Pipeline

① interpret  LLM (JSON-schema structured output) → QueryPlan  (fallback: keyword parser)
② finalizePlan  Zod validate · caller-override · per-viz constraints · status gate
③ buildQueryParams  filters → CT.gov v2 (query.* + Essie filter.advanced)
④ fetchStudies  paginate ALL matches · retry/backoff · totalCount
⑤ normalize  raw record → flat NormalizedStudy
⑥ aggregate  count / time-series / histogram / scatter / network  + citations
⑦ assemble  VisualizationSpec + meta + references[] + trace

■ LLM step (one only)    ■ deterministic code

The QueryPlan contract

LLM output is constrained to a closed schema (enums), then re-validated with Zod. It describes how to query — never the data.

  • controlled vocabularies → no invented fields
  • status, visualizationType, filters, groupBy, sort/topN, comparison, network
{
  "status": "ok",
  "visualizationType": "bar_chart",
  "filters": { "condition": "melanoma" },
  "groupBy": "phase",
  "confidence": 0.9,
  "notes": "..."
}

Anti-hallucination & safety guards

① LLM can't produce data — only a plan

② Structured output + Zod re-validation

③ Constraint layer — repairs/downgrades bad plans

④ Status gate — refuses unanswerable questions

⑤ Exact counts — full fetch, honest limits

⑥ Full traceability — apiRequests + citations

Validation happens at the tool-call layer (JSON schema) and in app code (Zod) — defense in depth.

"Is a chart even possible?" — the status gate

statuswhenresponse
okanswerablevisualization populated
needs_clarificationtoo vague / zero filtersnull + ask for specifics
unsupportedout of scope (patient counts, prices)null + explain why

A deterministic guard downgrades a zero-filter "ok" plan → needs_clarification: prevents both a meaningless answer and a 600k-record runaway fetch.

Data layer: exact & traceable

Full fetch by default

  • paginate ALL matches → exact counts (truncated=false)
  • retry/backoff, 30s timeout, totalCount
  • optional MAX_STUDIES cap (then honestly flagged)

Deep citations + references

  • every datum → {nct_id, excerpt, url}
  • references[] source cards (title, sponsor, PI, link)
  • trace: QueryPlan + raw API sample

Every bar/edge can be traced back to the exact trial records that produced it.

Resolving multi-signal queries

"Compare Drug A vs B over time" — which signal wins?

Not "one wins" — they compose. time_series (continuous axis) + comparison as series → a line per drug.

Precedence: network > time (composes with compare) > compare > distribution. Ranking is a modifier on bar_chart, not a type.

Network wins first (different data shape). Time absorbs comparison as series because its axis is continuous.

Observability & evaluation

Langfuse

LLM trace per request — prompt, tokens, cost, latency auto-captured.

Deterministic quality scores (fallback rate, empty result, confidence) + sampled LLM-as-judge.

OpenTelemetry → Jaeger

Span tree per request: interpret → fetch (per page) → aggregate.

Shows where latency goes + external HTTP calls.

Both optional (no-op without keys). Langfuse = LLM quality/cost · OTel = system performance.

Engineering quality

  • Unit tests — Vitest, 42 cases on the deterministic core (aggregators, planner guards, queryBuilder, normalize, schema)
  • Type safety — strict TS, single source of truth in types.ts
  • OpenAPI 3.1 spec — verified against live responses
  • Docker — multi-stage build
  • Prompt eval — promptfoo regression suite

Tradeoffs & what's next

  • Full fetch is slow for huge queries → parallelize pagination, or per-bucket countTotal for counts + sample records for citations
  • Entity normalization — "Merck" vs "MSD" are distinct nodes → alias resolution
  • No cache — short-TTL cache keyed by query params
  • scatter rarely auto-selected → stronger prompt cues

Designed deliberately: plan/data separation, the QueryPlan contract + constraint layer, citation model, CT.gov query mapping (verified live).
